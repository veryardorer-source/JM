# JM Notion System Handoff for Claude

작성일: 2026-06-16

## 0. 가장 중요한 주의사항

- Notion API 토큰을 문서나 스크립트에 직접 적지 말 것.
- 기존 작업 파일에는 토큰이 하드코딩되어 있었음. 다음 작업 전 토큰을 환경변수나 별도 비공개 파일로 옮기는 것이 우선.
- 전 직원이 보는 `진행중인 현장`에는 금액, 경비 합계, 출금 합계, 총지출, 계약금액, 고객 연락처가 노출되면 안 됨.
- Notion은 같은 DB 안에서 열 단위 권한 분리가 불가능함. 민감정보는 반드시 별도 관리자 DB/페이지에 둬야 함.
- 모바일 사용성이 최우선임. 현장팀은 대부분 휴대폰으로 확인하므로 표(table) 중심 화면을 만들면 안 됨.

## 1. 사용자의 최종 요구

회사는 Notion을 다음 용도로 사용하려고 함.

1. 현재 진행 중인 프로젝트 확인
2. 각 프로젝트의 현장 자료 공유
3. 시공 공정 단계와 예정일 확인
4. 블로그 포스팅 계획/실행 체크
5. 인스타 계획/실행 체크
6. 직원 정보 관리, 관리자만
7. 회사 매출/매입/고정비/현장별 비용, 관리자만
8. 고객 DB, 관리자만
9. JM에서 진행한 공사 리스트, 관리자만
10. 회사 공지사항 및 신입용 매뉴얼
11. 디자인팀 매뉴얼
12. 시공팀 매뉴얼
13. 현장팀 카드 지출 영수증 업로드, 현장팀+관리팀
14. 출금 요청 업로드, 현장팀+관리팀

## 4. 현장 DB 최종 방향

`진행중인 현장`은 전 직원용 현장 자료 허브다.

이 DB에는 돈 관련 열을 넣으면 안 된다.
사진과 자료 파일도 속성칸이나 현장 페이지 본문에 직접 계속 늘리지 말고, 각 현장 페이지의 하위 사진첩 페이지에서 관리한다.

최종 확인된 전 직원 현장 DB에는 민감 열이 제거되어 있었음:

- 경비합계 없음
- 출금합계 없음
- 총지출 없음
- 경비/출금/재무/회계 역방향 relation 없음

남겨도 되는 연결:

- 자재
- 공정표
- 업무
- A/S
- SNS

## 5. 완공 후 자료 흐름

현장이 끝나도 자료는 이동하거나 삭제하지 않는다.

분류 방식:

- 같은 `진행중인 현장` DB 안에 계속 보관
- `진행단계`를 `완공`으로 바꾸면 보기 필터로 자동 분리

보기 구성:

- `현장 갤러리`: 완공 제외, 진행 중 현장만 표시
- `완료 현장`: 완공만 표시, 완성 사진을 카드 표지로 사용

사진 필드는 다음 3개로 나뉘어야 함:

- `시공전 사진`
- `시공중 사진`
- `완성 사진`

최종 운영 기준은 DB 속성 3개나 본문 직접 나열이 아니라, 현장 페이지 아래의 `시공전 사진첩 / 시공중 사진첩 / 완성 사진첩` 하위 페이지다. 기존 `시공전 사진` 속성은 삭제 완료했고, 새 사진은 사진첩 하위 페이지에 올린다.

이 구조는 블로그 3편 구성과도 연결됨:

- 디자인/설계 편: 도면, 3D, 컨셉 사진
- 시공 편: 시공 사진, 공정표
- 마감 편: 시공전 사진, 완성 사진

## 6. 모바일 사용성 기준

현장팀은 휴대폰으로 바로 확인해야 한다.

원칙:

- 표(table) 중심 화면 금지
- 갤러리, 보드, 리스트 중심
- 속성명 앞 이모지 제거
- 모바일에서 잘리는 긴 이름 최소화
- 새 항목 추가가 쉬워야 함

이미 정리된 보기:

- `진행중인 현장`: 갤러리 보기
- `완료 현장`: 갤러리 보기
- `공정 일정`: 상태별 보드
- `업무 현황`: 진행상태별 보드
- `현장 경비/영수증`: 영수증 사진 갤러리
- `출금 요청`: 상태별 보드
- 메인 대시보드: `notion_decorate.py`로 직원 모바일 안내, 사진 삭제 안내, 폴더 바로가기 보강 예정
- 현장 카드 사진: `notion_site_album_pages.py`로 `시공전 사진첩 / 시공중 사진첩 / 완성 사진첩 / 도면·3D / 고객 자료` 하위 페이지 적용

아직 모바일 정리가 덜 된 영역:

- SNS/블로그 발행
- 자재/발주
- A/S
- 협력업체
- 공지/매뉴얼
- 직원 명부
- 근태
- 매출/매입
- 고객 CRM
- 견적
- 공사 아카이브

이 영역들도 모바일에서 쓸 가능성이 있다면 표를 기본으로 두지 말고 보드/리스트/갤러리 보기로 바꿔야 함.

## 7. 현재 발견된 문제

### 문제 1. 토큰 노출

Notion API 토큰이 여러 Python 파일과 `STRUCTURE.md`에 직접 적혀 있었음.

조치 필요:

- 토큰을 코드와 문서에서 제거
- 환경변수 사용 권장
- 공유 문서에는 절대 토큰 기록 금지

### 문제 2. 문서 설명 충돌

`STRUCTURE.md`에는 예전 자동집계 구조와 이후 권한 분리 수정 내용이 함께 남아 있음.

최종 기준은 권한 분리 수정 후 상태임.

즉:

- 전 직원 현장 DB에서 경비/출금/총지출 자동집계는 제거됨
- 관리자 `현장별 재무`의 `총지출`은 현재 수동 입력 방식
- `수익 = 계약금액 - 총지출`

문서에서 예전 설명은 “옛 구조”로 확실히 분리하거나 삭제해야 함.

### 문제 3. 새 시스템과 옛 시스템 혼재

옛 HUB가 아직 남아 있음.

직원 배포 전 해야 할 일:

- 새 `JM 업무 시스템`을 사이드바 최상위로 이동
- 옛 HUB는 백업 폴더로 이동하거나 이름에 `백업` 표시
- 직원들이 옛 시스템으로 들어가지 않게 정리

### 문제 4. 관리자 재무 자동화 약화

권한 보호 때문에 전 직원 현장 DB에서 자동 총지출을 제거했음.

현재는 관리자 `현장별 재무`에서 총지출을 직접 입력해야 함.

장점:

- 전 직원에게 금액 노출 안 됨

단점:

- 재무 자동집계 약화
- 관리자가 경리나라/경비/출금 내역을 보고 총지출을 업데이트해야 함

향후 개선하려면 별도의 관리자 전용 현장 재무 DB를 만들어 경비/출금과 연결하는 방식이 필요함.

## 8. 절대 되돌리면 안 되는 결정

- 전 직원 현장 DB에 금액 관련 열을 다시 넣지 말 것.
- 경비/출금/재무 DB를 전 직원 현장 DB와 양방향 relation으로 다시 연결하지 말 것.
- 고객명/연락처를 전 직원용 현장 DB에 넣지 말 것.
- 모바일 대시보드를 넓은 표 보기 중심으로 되돌리지 말 것.
- 사진 필드를 다시 `현장 사진` 하나로 합치지 말 것. `시공전/시공중/완성` 3분류 유지.
- 완료 현장을 별도 DB로 복사/이동하지 말 것. 같은 현장 DB 안에서 `진행단계`와 보기 필터로 분류하는 것이 현재 기준.

## 9. 클로드가 다음에 하면 좋은 작업

우선순위 순서:

1. `notion_preflight_audit.py`로 새 시스템 읽기 전용 점검
2. `notion_decorate.py`로 메인 대시보드 모바일 안내/사진 삭제 안내/바로가기 적용
3. 새 시스템을 사이드바 최상위로 이동
4. 옛 HUB 백업 처리
5. 남은 DB 모바일 보기 정리
6. 관리자 전용 재무 자동화 재설계 검토
7. 직원/현장팀/관리자 권한 초대 실제 적용
8. 매뉴얼 내용 채우기

토큰 제거, 환경변수화, `STRUCTURE.md` 정리는 로컬 파일 기준 완료됨. 단, 실제 Notion 실행 전에는 새 토큰을 현재 PowerShell 창의 `NOTION_TOKEN`에만 넣고 `notion_preflight_audit.py`를 먼저 실행해야 함.

사진은 각 사진첩 하위 페이지에서 직원이 직접 파일 메뉴로 삭제할 수 있어야 함. 전 직원 현장 DB에 사진 파일 속성을 다시 만들지 말 것.

## 10. 사용자에게 설명할 때 쓰면 좋은 요약

현재 시스템은 크게 맞게 만들어졌지만, 배포 전 정리가 필요하다.

현장팀은 모바일에서:

- 현장 카드 보기
- 공정 상태 보기
- 업무 상태 보기
- 영수증 사진 올리기
- 출금 요청 올리기

를 할 수 있게 되어 있다.

전 직원 현장 화면에는 돈 관련 정보가 빠져 있고, 돈은 관리자 전용 영역에서만 관리한다.

완공된 현장은 삭제되거나 이동되는 것이 아니라, 같은 현장 DB 안에서 `완공` 단계로 바뀌며 `완료 현장` 보기로 자동 분류된다.
